# deploychan_mcp

MCP-сервер «опыт вайбкода KISA». Публичный опенсорс remote-MCP, который отдаёт
чужим кодинг-агентам курированный вайбкод-опыт автора: знания, готовые паки,
маршруты прокачки и вкус. Этот файл — **источник истины по продукту и решениям**.
Держи его в актуальном состоянии; всё остальное в репо подчиняется ему.

> Стадия: **сырая идея → скелет.** Бэкенд намеренно описан как *направление*, а не
> жёсткая спека — финальная форма уточнится после того, как будет готов визуальный
> дизайн сайта (его делает оператор в claude design). Фронт ведёт, бэк подстраивается.

---

## TL;DR

- **Продукт.** Удалённый MCP-сервер. Любой вставляет URL в своего агента и
  прокачивает его опытом KISA.
- **Сайт `mcp.deploychan.webcam`.** Витрина (каталог контента) + инструкция «как
  подключить». Делается **отдельно** в claude design как React-сайт; для правок вне
  дизайна в него встраивается `agentation` (см. «Дизайн сайта и совместимость»).
- **Доступ.** Публично, без логина.
- **Контент.** Ручная курация — оператор пишет руками в `content/`. Никакого
  скрапинга YouTube/TG на MVP.
- **Стек бэка.** Python + официальный MCP SDK (FastMCP), транспорт Streamable HTTP,
  хранилище SQLite + FTS5. Чистый самостоятельный репо, без зависимости от демона
  Deploychan.
- **Лицензия.** MIT. Репо публичный: никакой приватной информации внутри.

## Что это и зачем

KISA годами оставляет опыт вайбкода на YouTube и в Telegram. Идея — упаковать этот
опыт так, чтобы **чужой агент мог его «съесть» и стать лучше**. Канал доставки —
MCP: агент подключается к серверу и через инструменты тянет знания, готовые паки и
маршруты прокачки.

Это **не** локальный демон `Deploychan` (тот гейтит действия агента по плану).
Здесь обратная задача: публичный сервер **раздаёт** знание, а не ограничивает.
Общее — только паттерны инфраструктуры (SQLite+FTS5, идея wiki-адаптера), не код.

## Три роли

| Роль | Кто | Что делает |
|---|---|---|
| **Агент-потребитель** | чужой кодинг-агент (Claude Code, Cursor, …) | Подключается по MCP URL, дёргает инструменты, получает контекст / паки. |
| **Посетитель-человек** | разработчик на сайте | Понимает ценность, копирует URL подключения, листает каталог. |
| **Оператор** | KISA | Наполняет `content/`, билдит каталог, деплоит сервер. |

## Решения (зафиксированы совместно, 2 раунда опроса)

| Тема | Решение |
|---|---|
| Связь с демоном Deploychan | Отдельный продукт, **общая инфра как образец** (SQLite+FTS5, идея wiki-адаптера), без импортов. |
| Что MCP отдаёт агенту | Четыре грани: **Знания** (ретрив), **Скиллы** (готовые паки), **Прокачка** (маршруты + советы как настроиться), **Рекомендуемое** (стартовая подборка). |
| Доступ | **Публично, без логина.** Скопировал URL — подключил. |
| Источник данных | **Ручная курация.** Контент пишется руками в `content/`. Скрапинг YouTube/TG — не в MVP. |
| Транспорт | **Remote HTTP (Streamable).** URL вида `https://mcp.deploychan.webcam/mcp`. |
| Стек бэка | **Python, чистый репо.** MCP SDK (FastMCP) + своя БД. |
| Набор инструментов | `search_knowledge`, `list_skills`/`get_skill`, `onboard`/`next_step`, `list_recommended`. |
| Сайт | **Лендинг + как подключить** и **каталог контента** (фасеты Знания/Скиллы/Прокачка/Рекомендуемое). Фронт оператор делает с нуля в claude design. Плейграунд/живая демка — не в MVP. |
| Открытость | **Опенсорс (MIT).** В репо ноль приватных данных. |

## MCP-инструменты — контракт

Четыре инструмента, по одному на грань каталога. Сигнатуры провизорные: финал
утвердим после визуала, но смысл фиксирован.

| Инструмент | Сигнатура (черновик) | Грань | Что возвращает |
|---|---|---|---|
| `search_knowledge` | `(query: str, limit: int = 5)` | Знания | FTS5-куски из `content/knowledge/`: `title`, `snippet`, `source`. |
| `list_skills` / `get_skill` | `()` / `(skill_id: str)` | Скиллы | Каталог скиллов → полный скилл для установки (шаги, конфиг, чеклист). |
| `onboard` / `next_step` | `(goal: str)` / `(route_id: str)` | Прокачка | Маршрут под цель (упорядоченные шаги-ссылки на другие айтемы) → материалы и действия шага. |
| `list_recommended` | `()` | Рекомендуемое | Кураторская стартовая подборка KISA (кросс-тип). |

**Онбординг — stateless.** Логина нет → серверной сессии нет. Прогресс держит
клиент: `onboard` отдаёт весь маршрут со списком шагов-id, `next_step(route_id)`
отдаёт материалы и действия конкретного шага. Никакого состояния на сервере per-user.

## Контент-модель

Источник истины по контенту — **файлы в `content/`** (markdown + YAML-frontmatter).
Билд-шаг (`ingest`) загружает их в SQLite, строит FTS5-индекс и `web/catalog.json`.
Каталог размечен по **двум осям** + флаги.

**Ось 1 — тип контента** (он же папка):

| Папка | Тип | Идёт в инструмент |
|---|---|---|
| `content/skills/<id>/SKILL.md` | `skill` | `list_skills` / `get_skill` |
| `content/knowledge/<id>.md` | `knowledge` | `search_knowledge` |
| `content/routes/<id>.md` | `route` (Прокачка) | `onboard` / `next_step` |
| `content/tools/<id>.md` | `tool` (гайд по установке стороннего) | каталог / `search_knowledge` |

**Ось 2 — авторство:** `author: kisa` или `author: third_party` (бейдж «сторонний»).

**Флаги (фронтматтер любого айтема):** `recommended: true` (★ → `list_recommended()`),
`added: <дата>`, `tags: [...]`.

- **Поиск на MVP — только FTS5.** Без эмбеддингов и внешних API: оффлайн, без
  ключей, безопасно для публичного репо. Семантический/векторный поиск — апгрейд позже.
- **Контент = публичный.** Всё, что лежит в `content/`, видно сообществу. Приватное
  сюда не кладём.

## Сайт — задача и контракт для claude design

Фронт делает оператор с нуля в claude design. Этот раздел — то, что claude design
должен знать, чтобы выдать совместимый результат.

**Задачи сайта (MVP):**
1. **Лендинг + «как подключить»** — что это, кому, зачем; копипаст MCP URL;
   сниппеты конфига под Claude Code / Cursor / прочих клиентов.
2. **Каталог** — браузабельная витрина контента с фасетами `01–04`: Знания / Скиллы /
   Прокачка / Рекомендуемое. Фильтры по типу и по авторству (KISA / сторонний); бейджи
   ★ / «сторонний» / дата.

**Структура доков:** Введение → Быстрый старт → 4 инструмента (Знания, Скиллы,
Прокачка, Рекомендуемое) → Установка MCP (селф-хостинг) → Безопасность.

**Контракт данных.** Фронт **статичный**, в браузере к MCP не ходит (плейграунд не
в MVP). Ему нужны два артефакта, оба генерит бэк из `content/`:

- `web/catalog.json` — витрина:
  ```jsonc
  {
    "items": [
      { "id": "...", "name": "...", "summary": "...",
        "type": "skill|knowledge|route|tool",
        "author": "kisa|third_party",
        "recommended": true, "added": "2026-06-26", "tags": ["..."] }
    ],
    "connect": { "url": "https://mcp.deploychan.webcam/mcp",
                 "clients": { "claude-code": "...", "cursor": "..." } },
    "profile": { "name": "KISA", "links": { "youtube": "...", "telegram": "..." } }
  }
  ```
- Сниппеты «как подключить» — лежат в `catalog.json.connect` (дублируются в README).

## Дизайн сайта и совместимость с agentation

**Уточнение (проверено по факту, не по памяти).** Инструмент, которым собраны
лендинг и kisa.site, называется **`agentation`** (npm-пакет; `agentation.dev`,
`github.com/benjitaylor/agentation`, лицензия PolyForm-Shield). Это **не стек и не
генератор сайтов**, а dev-time React-компонент — дословно «Visual feedback for AI
coding agents»: рендерит тулбар, оператор кликает по элементам и пишет заметки, а
компонент отдаёт агенту точные селекторы/позиции (или синкает на локальный сервер
через `endpoint`).

**Единственное жёсткое требование совместимости — сайт должен быть React-приложением
(React ≥ 18).** Больше `agentation` ничего не диктует (`peerDependencies: react >=18`,
zero runtime deps). Подтверждено: в проектах оператора он стоит в ДВУХ разных стеках —
- `../deploychan-landing` — no-build: React 18 UMD + Babel standalone, `content.js` +
  один `*.jsx`; `agentation` грузится с esm.sh только на localhost;
- `../kisa.deploychan.webcam/kisa-site` — Next.js 16 + React 19 + Tailwind + Zustand;
  `agentation` как dev-зависимость (`components/AgentationDev.tsx`).

Поэтому claude design может отдать **любой React-вариант** — он останется
редактируемым через agentation. (Прежняя формулировка «формат = React18 UMD + Babel»
была слишком узкой — это стек лендинга, а не требование инструмента.)

**Как встроить agentation (dev-only; обе формы рабочие, взяты из реальных проектов):**

Framework — если claude design отдаёт Next/Vite (как kisa-site):

```tsx
// components/AgentationDev.tsx — рендерить только в dev
import { Agentation } from "agentation";
export function AgentationDev() {
  if (process.env.NODE_ENV !== "development") return null;
  return <Agentation endpoint="http://localhost:4747" />;
}
```

No-build — single-file (как лендинг), грузится только на localhost:

```js
if (['localhost','127.0.0.1'].includes(location.hostname)) {
  const [{ default: R }, { createRoot }, { Agentation }] = await Promise.all([
    import('https://esm.sh/react@18.3.1'),
    import('https://esm.sh/react-dom@18.3.1/client'),
    import('https://esm.sh/agentation@3.0.2?deps=react@18.3.1,react-dom@18.3.1'),
  ]);
  const mount = document.createElement('div');
  document.body.appendChild(mount);
  createRoot(mount).render(R.createElement(Agentation, { endpoint: 'http://localhost:4747' }));
}
```

`endpoint: http://localhost:4747` — опциональный «Agent Sync» (локальный сервер,
мостящий фидбэк в кодинг-агента). Без него `agentation` просто копирует
структурированный markdown в буфер.

**Рекомендация для этого сайта — no-build single-file** (как лендинг): `index.html` +
`content.js` (тексты/данные как `window`-глобалы) + один `*.jsx` (CSS в строке `*_CSS`,
темы через `[data-theme]`) + `serve.json`. Причина: минимум трения, оператор уже
редактирует лендинг ровно так, проще всего «дорабатывать вне claude design». Если
claude design отдаст Next/Vite — тоже валидно, просто добавляем `agentation` как
`-D`-зависимость.

**Ожидаемые `content.js`-глобалы (для no-build варианта):**

| Глобал | Что |
|---|---|
| `HERO` | заголовок/подзаголовок, оффер «прокачай агента». |
| `CONNECT` | MCP URL + сниппеты подключения по клиентам. |
| `CATALOG` | айтемы каталога (skills/knowledge/routes/tools, из `web/catalog.json`). |
| `PROFILE` | бренд KISA: имя, био, ссылки YouTube/TG. |
| `FAQ` | частые вопросы (что это, безопасно ли, бесплатно ли). |

**Что сказать claude design:** «отдай React-сайт (≥18); я встрою `agentation` для
правок». Дальше — single-file предпочтительно, framework допустимо.

**Тон/бренд** — в духе kisa.site: моно-шрифты, терминальная эстетика, характер.
Палитры — Catppuccin (Latte/Mocha) + Tokyo Night / Rosé Pine (см.
`../deploychan-landing/palettes.html`).

## Бэкенд — направление (провизорно, финал после визуала)

Не реализуем жёстко сейчас. Зафиксированное направление:

```
deploychan_mcp/
├── server/
│   ├── app.py       # FastMCP-сервер, Streamable HTTP, регистрация инструментов
│   ├── tools.py     # 4 инструмента (см. контракт выше)
│   ├── db.py        # SQLite + FTS5: схема, миграции
│   ├── ingest.py    # content/ → БД + генерация web/catalog.json
│   └── models.py    # dataclasses: Doc / Pack / Route
├── content/         # курируемое знание (заполняет оператор)
├── web/             # фронт из claude design + сгенерённый catalog.json
├── tests/
├── pyproject.toml
├── README.md        # публичное лицо: что это, quickstart, как подключить
└── LICENSE          # MIT
```

- **Транспорт:** Streamable HTTP (современный стандарт remote-MCP).
- **Хранилище:** SQLite + FTS5; таблица `items` (поля `type`/`author`/флаги) + FTS-вьюха
  над текстом. Паттерн взят как образец из `Deploychan/deploychan/db.py`, но код
  самостоятельный.
- **Без состояния per-user.** Публичный доступ, онбординг stateless.

## Приватность и опенсорс

Репо **публичный**. Жёсткие запреты на содержимое:

- ❌ Пути к LLM Wiki / Obsidian vault, личные абсолютные пути.
- ❌ Токены, ключи, секреты, `.env` с реальными значениями.
- ❌ Приватные заметки, закрытые проекты, личные глобальные инструкции.
- ✅ Только курируемый публичный контент в `content/` и пример-плейсхолдеры.

Перед каждым коммитом — проверка, что в диффе нет приватного.

## Открытые вопросы / отложено

- **Домен.** Использую `deploychan.webcam` (везде в инфре `chan`, не `cham`) —
  подтвердить.
- **Хостинг.** Где крутится сервер и сайт — определить (вероятно, рядом с прочими
  `*.deploychan.webcam`).
- **Семантический поиск.** Эмбеддинги/вектор — апгрейд после MVP.
- **Биллинг/тиры.** Сейчас всё бесплатно и без логина; платное — позже.
- **Наполнение.** Реальный контент в `content/` — отдельная сессия после скелета.
- **Финал бэкенда.** Сигнатуры инструментов и схема БД утверждаются после готового
  визуала сайта.

## Глоссарий

- **MCP** — Model Context Protocol; способ подключить инструменты к агенту.
- **Скилл** — готовый набор (шаги установки/настройки/конфиг), который агент ставит себе.
- **Прокачка** — пошаговый маршрут под цель (`onboard`/`next_step`); вобрала прежний «вкус/мнение».
- **Сторонний** — ось авторства: контент, который KISA не разрабатывает (бейдж).
- **agentation** — npm-пакет «Visual feedback for AI coding agents» (оператор
  произносит «Agentstation»). Dev-time React-компонент `<Agentation/>` для
  визуального фидбэка агенту; требует только React ≥ 18. Не стек и не генератор.
  Образцы — `../deploychan-landing` и `../kisa.deploychan.webcam/kisa-site`.
- **Deploychan (демон)** — отдельный продукт: локальный гейт действий агента. Не
  путать с этим MCP-сервером.
